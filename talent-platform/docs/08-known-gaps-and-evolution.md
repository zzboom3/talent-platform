# 已知缺口、实现偏差与演进建议

这份文档专门记录项目当前实现中的不一致、未完成能力、潜在风险和适合继续优化的方向。

它的目的不是否定项目，而是让读者准确理解：

- 代码已经做到哪里
- 展示材料说到了哪里
- 哪些点是天然的优化空间

## 1. 文档与代码实现不完全一致的地方

## 1.1 “PDF 导出”目前更像打印

README 中提到“评估历史记录 + PDF 导出”，但当前前端实现更接近浏览器打印：

- 页面按钮文案为“导出/打印报告”
- 实现方式是 `window.print()`

这意味着：

- 能导出成浏览器打印结果
- 但不是成熟的服务端或前端 PDF 文件生成方案

## 1.2 “区块链浏览器”代码存在，但主路由没有入口

前端存在 `BlockchainExplorer.vue`，但当前主路由里并没有把它暴露出来。

当前真正能被用户访问到的区块链展示入口主要是：

- 证书详情页
- 管理后台区块链页面

因此文档中不应把“独立前台区块链浏览器”写成已完整对用户开放的主能力。

## 1.3 政策采集未真正接入来源

爬虫服务中政策抓取当前是占位实现，消息明确说明：

- 当前未配置政策源
- 新增 0 条待审核内容

所以当前“政策”更多依赖：

- 手工录入
- 或统一内容体系中的现有数据

## 1.4 展示材料比真实实现更完整

README、PPT 材料和答辩截图强调了很多亮点功能，这对展示很好，但技术文档必须基于源码现状而不是只基于展示话术。

## 2. 权限与安全层面的风险

这是当前项目最值得认真记录的一类问题。

## 2.1 不是所有写接口都做了严格角色限制

一些控制器主要依赖“需要登录”，但没有严格校验“必须是正确角色”。

例如：

- 企业资料创建接口并未强制要求一定是 `ENTERPRISE`
- 人才档案创建接口并未强制要求一定是 `TALENT`
- 课程管理类接口没有明显管理员限制

这意味着：

- 前端页面路径看起来区分了角色
- 但后端有些地方仍可能被越过页面直接调用

## 2.2 部分更新接口存在越权风险

基于源码梳理，下面几类接口尤其值得后续重点复核：

- 申请状态更新接口
- 学习进度更新接口
- 证书生成接口
- 测评详情按 ID 查询接口

这些接口如果缺少“资源归属校验”，就可能允许用户操作不属于自己的数据。

## 2.3 公开接口范围比直觉更宽

某些接口虽然名字偏管理或偏系统，但配置为公开 `GET`：

- 部分区块链读取接口
- 部分统计接口

这不一定是错误，但应该在文档里明确说明，避免误判安全边界。

## 2.4 JWT secret 使用默认配置值

当前 `application.yml` 中存在固定默认 JWT secret。

对于演示和开发环境，这是方便的；但对于正式环境，这显然应该改成：

- 更安全的外部化配置
- 更明确的环境区分

## 3. 工程化层面的缺口

## 3.1 没有数据库迁移工具

当前数据库结构主要依赖：

- JPA 实体
- `ddl-auto: update`

优点：

- 开发快
- 演示方便

缺点：

- 结构演进不透明
- 版本化能力弱
- 生产变更风险高

## 3.2 没有 OpenAPI / Swagger

这使得接口文档必须：

- 靠源码阅读
- 靠前端 `api/index.js`
- 靠手工整理

对学习者来说反而是很好的练习材料，但对团队协作不够友好。

## 3.3 测试体系很薄

目前只看到一个基础的启动测试。

缺失的测试包括：

- 控制器接口测试
- 权限回归测试
- 服务层逻辑测试
- AI 结果解析测试
- 上传与区块链流程测试

## 3.4 异常处理不统一

项目没有完整的全局异常处理器。

当前常见做法是：

- 在控制器里返回 `Result.fail(...)`
- 由 Security 统一处理 `401/403`

这种模式对小项目足够直接，但对长期维护不够稳定。

## 3.5 DTO 使用不系统

当前接口大量直接使用：

- 实体对象
- `Map<String, Object>`

好处是写起来快，坏处是：

- 请求结构不够明确
- 验证与演进成本更高

## 3.6 参数校验未充分启用

虽然依赖里引入了 Validation，但源码里没有形成系统的 `@Valid` + 校验注解体系。

这意味着很多参数约束实际上靠：

- 前端表单
- 控制器手工判断

## 4. 架构层面的真实取舍

## 4.1 Controller 过重

项目里很多控制器不只是参数转发层，还承担：

- 权限判断
- 业务判断
- 仓储调用
- 响应拼装

### 评价

对中小型演示项目来说，这种方式很常见、开发效率高。

但随着业务增多，会出现：

- 控制器越来越长
- 逻辑复用困难
- 安全检查分散

## 4.2 领域对象与展示对象没有明显分层

很多接口直接返回实体对象，这带来两个结果：

- 前后端联调快
- 接口边界和实体边界耦合较强

当以后要演进接口时，DTO 缺失会增加成本。

## 4.3 状态建模不统一

一部分状态是枚举，一部分是字符串。

这意味着长期维护中很容易出现：

- 拼写不一致
- 状态扩展缺乏编译期保护

## 5. 产品层面的已知偏差

## 5.1 招聘模型双线并存

项目同时拥有：

- `Job`
- `TalentRecruitment`

如果不在文档中明确解释，新读者会不清楚：

- 哪个是主招聘实体
- 哪个是扩展招聘需求

## 5.2 用户旅程与权限旅程并不完全一致

从产品角度看，角色是明确的：

- 人才做人才的事
- 企业做企业的事
- 管理员做后台运营

但从技术实现角度看，部分页面和接口没有做到完全严密的角色封闭。

这就是“产品意图”和“代码边界”的差距。

## 5.3 后台接口体验依赖前端特殊约定

后台管理之所以不会被全局 401 自动踢下线，很大程度依赖前端 `skipAuthRedirect: true` 约定。

这很好用，但也意味着：

- 新增后台接口时很容易遗漏
- 一旦遗漏，体验会比较差

## 6. 哪些点最适合作为后续优化方向

## 6.1 安全与权限优先级最高

建议优先做：

1. 逐接口补 owner/admin 校验
2. 严格限制角色创建和修改各自资料的边界
3. 收紧公开接口范围
4. 把关键安全配置迁移到环境变量

## 6.2 工程化第二优先级

建议逐步补：

1. 全局异常处理器
2. DTO 分层
3. Validation 校验
4. 数据库迁移工具
5. API 文档

## 6.3 测试体系第三优先级

建议至少补：

- 登录注册测试
- 企业审核测试
- 投递状态流转测试
- 学习到发证测试
- 权限回归测试

## 6.4 产品体验优化方向

可以考虑：

- 完善真正的 PDF 导出
- 给 `BlockchainExplorer` 增加正式入口
- 明确 `Job` 与 `TalentRecruitment` 的产品定位
- 完善角色页面兜底
- 更完整地展示 AI 结果来源和解释

## 7. 对毕业设计/答辩的价值

这些缺口不一定是坏事。对毕业设计或课程项目来说，它们反而可以成为：

- 系统分析的亮点
- 答辩时展示“我知道项目做到哪里、还差什么”的依据
- 后续优化章节的素材

如果答辩老师问“你们的项目还有哪些改进空间”，这份文档就是最好的答案库。

## 8. 推荐如何使用这份文档

### 如果你是开发者

把它当成风险清单和改造清单。

### 如果你是学习者

把它当成“真实项目为什么和教科书不同”的案例库。

### 如果你是答辩讲解者

把它当成“项目成熟度分析”和“后续演进方向”的素材库。

## 9. 最终结论

`talent-platform` 的完成度已经足以支撑：

- 完整产品演示
- 前后端联调学习
- 架构讲解
- 毕业设计答辩展示

但它同时也保留了很多典型的真实工程问题：

- 权限边界不完全统一
- 接口与模型分层不充分
- 文档和展示材料存在轻微超前表述
- 工程基础设施仍有提升空间

正因为如此，它既适合“展示成果”，也适合“研究怎么继续把项目做得更好”。
